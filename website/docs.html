Get started with tunnelX

Expose a local port to a secure public HTTPS URL in one command. This guide walks you through installation, everyday usage, configuration, and fixes for the most common issues.

Introduction

tunnelX is a lightweight, secure, and fast localhost tunneling tool. It gives your local web app a public HTTPS URL — https://my-app.darsha.dev → localhost:3000 — so you can share work, test webhooks, or demo to clients without deploying or touching your router.

The client is a single Rust binary. Traffic flows from the browser to a relay over HTTPS, then down to your machine over an encrypted WebSocket:

architecture
Browser → HTTPS → Relay Server → WSS → Tunnel Client → localhost

Installation & Setup

1. Prerequisites

tunnelX is built with Rust, so you need the Rust toolchain (which ships with cargo). If you don't have it yet, install it with rustup:

bash
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh

Confirm the toolchain is available:

bash
cargo --version   # cargo 1.7x or newer

2. Install the client

The quickest way is straight from crates.io:

bash
cargo install tunnelx

Or build from a local checkout of the repository:

bash
git clone https://github.com/Its-darshu/tunnelX.git
cd tunnelX
cargo install --path client

3. Verify the install

Cargo installs binaries to ~/.cargo/bin. Make sure that directory is on your PATH, then check the version:

bash
tunnelx --version   # tunnelx 0.1.0
Tip

If tunnelx: command not found appears, add cargo's bin directory to your shell profile: export PATH="$HOME/.cargo/bin:$PATH", then restart your terminal.

4. Run your first tunnel

Start a local server on any port (for example python3 -m http.server 3000), then expose it:

bash
tunnelx 3000

tunnelX prompts for a subdomain and duration, then prints your live URL:

output
🚀 TunnelX v0.1.0

[1/3] Establishing connection...
  ✔ Connected to TunnelX server
[2/3] Registering tunnel...
  ✔ Subdomain "my-app" registered!

🌐 Tunnel is live!

  https://my-app.darsha.dev  →  localhost:3000

📋 Share this URL with your friends!

5. Shell completions (optional)

Generate completion scripts for your shell so command and flag names auto-complete:

bash
# Bash
tunnelx completions bash > ~/.bash_completions
echo "source ~/.bash_completions" >> ~/.bashrc

# Zsh
tunnelx completions zsh > /usr/local/share/zsh/site-functions/_tunnelx

# Fish
tunnelx completions fish | sudo tee /usr/share/fish/vendor_completions.d/tunnelx.fish

Quick Start

The most common workflows in one place:

bash
# Interactive — prompts for subdomain and duration
tunnelx 3000

# Non-interactive — specify everything upfront
tunnelx tunnel 3000 --subdomain my-app --duration 1800

# Point at a custom relay server
tunnelx 3000 --relay wss://relay.example.com/tunnel

# List active tunnels
tunnelx list

# Check the status of one tunnel
tunnelx status my-app

While a tunnel is running you can use these keyboard shortcuts:

KeyAction
cCopy the public URL to your clipboard
qQuit and close the tunnel
Ctrl + CQuit and close the tunnel

Command Reference

tunnel — start a new tunnel

Expose a local port. When run as tunnelx <PORT> the tunnel subcommand is implied.

OptionDescription
-s, --subdomain <NAME>Requested subdomain (skips the interactive prompt)
-d, --duration <SECONDS>Tunnel duration in seconds
--relay <URL>Custom relay WebSocket URL
--profile <NAME>Use a specific configuration profile
--format <text|json>Output format (default: text)

config — manage configuration & profiles

bash
tunnelx config init                 # create ~/.config/tunnelx/config.toml
tunnelx config set <KEY> <VALUE>     # set a value
tunnelx config get [KEY]            # read all or one value
tunnelx config profiles            # list profiles
tunnelx config validate            # check the config is valid
tunnelx config reset --force       # restore defaults

list — list active tunnels

bash
tunnelx list              # simple list
tunnelx list --detailed   # full table view
tunnelx list --format json

status — show tunnel status

bash
tunnelx status my-app
tunnelx status my-app --format json

Global options

OptionDescription
-h, --helpShow the help message
-V, --versionShow the version
--debugEnable debug logging (also TUNNELX_DEBUG=1)

Configuration

tunnelX reads settings from ~/.config/tunnelx/config.toml. Profiles let you keep separate relay/duration defaults for, say, development and production.

config.toml
[profile.default]
relay = "wss://relay.darsha.dev/tunnel"
duration = 1800

[profile.production]
relay = "wss://prod-relay.example.com/tunnel"
duration = 3600

Environment variables

TUNNELX_DEBUG—Set to 1 to enable debug logging
RELAY_BIND0.0.0.0:8443Relay listen address (self-hosting)

Self-Hosting the Relay

You can run your own relay for local development — no TLS certificate required:

bash
RELAY_BIND=127.0.0.1:8443 RELAY_DOMAIN=localhost cargo run --package relay
tunnelx 3000 --relay ws://127.0.0.1:8443/tunnel
Heads up

Generated subdomains don't exist in public DNS during local development, so requests need a matching Host header. Replace <subdomain> with the value the client prints:

bash
curl -H 'Host: <subdomain>.localhost' http://127.0.0.1:8443/

For a production deployment with Docker, wildcard DNS, and TLS certificates for *.darsha.dev, see docs/deployment.md.

Troubleshooting

Most problems fall into one of the buckets below. Run any tunnel command with --debug to see verbose logs while diagnosing.

command not found: tunnelx

The binary installed but isn't on your PATH. Cargo installs to ~/.cargo/bin — add it to your shell profile and reload:

bash
echo 'export PATH="$HOME/.cargo/bin:$PATH"' >> ~/.bashrc
source ~/.bashrc
tunnelx --version

Connection / relay issues

If the tunnel can't establish a connection, confirm which relay it's targeting and re-run with debug logging:

bash
# See exactly what the client is doing
tunnelx tunnel 3000 --debug

# Check the configured relay URL
tunnelx config get profile.default.relay
Tip

A corporate proxy or firewall can block outbound WebSocket (wss://) traffic. Try from a different network to rule this out, or point --relay at a relay you control.

Local server not reachable through the tunnel

The tunnel connects but requests return errors. Check that your app is actually listening on the port you exposed and is bound to localhost/127.0.0.1 (not only an external interface). Confirm with:

bash
curl -I http://localhost:3000   # should return an HTTP status

Configuration problems

If a command complains about invalid or missing config, validate it — then reset if needed:

bash
tunnelx config validate       # report problems
tunnelx config path           # show which file is in use
tunnelx config reset --force  # restore defaults

Shell completions not working

Regenerate the completion script and re-source it (or open a new terminal):

bash
tunnelx completions bash > ~/.bash_completions
source ~/.bash_completions

Subdomain already taken

If a requested subdomain is in use, pick another with --subdomain, or omit it to get a random one. Check what's currently active with tunnelx list.
